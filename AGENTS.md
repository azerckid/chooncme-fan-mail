# Antigravity Global Standards & Communication Rules

1. **커뮤니케이션 및 승인 원칙**
   - 사용자와의 모든 대화에서 이모지(Emoji) 및 이모티콘 사용을 전면 금지한다. 전문적이고 명확한 텍스트로만 소통한다.
   - 질문이나 설명 요청 시, 파일 수정이나 명령 실행을 금지하고 상세한 텍스트 답변을 최우선으로 제공한다.
   - 코드를 수정하기 전에 반드시 수정 계획을 보고하고 명시적 승인을 얻어야 한다.
   - 코드 수정 지시가 아닌 일반적인 대화에는 응답만 수행하며, 임의로 코드를 건드리지 않는다.
   - 코드 수정을 시작할 때는 반드시 "OOO 기능을 수정하겠습니다"라고 목적을 명확히 밝힌다.
   - 사용자의 질문에 답변을 가장 먼저 완료하고, 다음 지시를 기다린다.
   - 문서에 정의되지 않은 작업 요청 시, 구현 전 문서 업데이트 필요성을 먼저 확인한다.
   - **[Evidence-Based Response]** 도구로 현재 상태를 검증한 후 답변해야 한다. 추측성 발언 금지.
   - **[Next-Step Simulation]** 문제 해결 후 후속 시나리오를 시뮬레이션하여 선제 조치한다.

2. **보안 및 데이터 안전**
   - 모든 DB 작업 전 반드시 전체 백업(Dump)을 수행한다.
   - 백업 확인 전 파괴적인 명령어 실행 금지.
   - .env* 파일의 Git 커밋 금지. .gitignore 선제 점검.
   - Local과 Production 환경 엄격 격리.

3. **개발 표준 및 품질**
   - 중요 작업 전 git commit 수행 여부 확인.
   - 표준 아키텍처(Best Practice) 준수.
   - 공통 로직 수정 시 영향 범위 분석 및 격리.
   - 속도보다 정확성 최우선.
   - 운영 환경(Production)을 전제로 사고 금지.

4. **문서 관리 표준**
   - 문서 수정 시 기존 구조 보존, 점진적 업데이트.
   - `docs/` 하위 365 Principle 5대 레이어 구조 엄격 준수:
     - `01_Concept_Design/`: 비전, 제품 기획, 비즈니스 전략
     - `02_UI_Screens/`: UI 디자인 시스템, 화면 흐름
     - `03_Technical_Specs/`: DB 스키마, API 명세, 개발 원칙
     - `04_Logic_Progress/`: 로드맵, 백로그, 구현 계획
     - `05_QA_Validation/`: 테스트 시나리오, 검증 보고서

5. **단계별 품질 보증 및 완료 조건**
   - 각 Phase 종료 전 통합 사용자 시나리오 테스트(E2E) 필수.
   - 브라우저 환경 검증 필수.
   - 완료 기준(DoD): No 404, No UI Error, Console Clean, Error Feedback.
   - 완료 보고서에 QA 결과 포함.

6. **코드 리뷰 및 리팩토링 표준**
   - 불필요한 주석/로그 제거, 타입 점검 후 보고.
   - 복잡한 로직은 설명 추가.
   - 비효율적 코드 발견 시 리팩토링 제안.
   - API/인증 수정 시 권한 검사 로직 재확인.

7. **배포 표준**
   - 운영 환경 배포 및 정상 작동 확인을 완료 기준으로 함.
   - 배포 전 DB 정합성 및 마이그레이션 상태 확약.
   - 배포 후 실서비스 URL에서 E2E 재점검.

8. **환경변수 관리 표준**
   - `.env.development`, `.env.production` 격리.
   - 시스템 환경변수는 호스팅 대시보드 관리.
   - 동적 구성(localhost vs domain).

9. **Git 커밋 컨벤션**
   - `type(scope): 설명` (예: `feat(db): 춘심이 팬레터 테이블 생성`)
   - Type은 영어(feat, fix 등), 설명은 한글 소문자, 3줄 이상 상세.
